# Implementation Plan: API Key Budget Enforcement & Spend Tracking

**Branch**: `009-key-budget-spend` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/009-key-budget-spend/spec.md`

---

## Summary

Enable LiteLLM's built-in virtual key budget system (already present but dormant in v1.52.0) by wiring `LITELLM_SALT_KEY` into the litellm service, then expose key management and spend reporting through Kong. Build a minimal portal-backend FastAPI service that aggregates LiteLLM's per-key and per-model spend data into a single `GET /v1/spend` response. Langfuse cost metadata requires no new code — the `langfuse` callback and `LANGFUSE_HOST` are already configured.

---

## Technical Context

**Language/Version**: Python 3.12 (portal-backend), YAML (LiteLLM config, Kong config)

**Primary Dependencies**:
- LiteLLM v1.52.0 — virtual key management + spend tracking (built-in, no upgrade)
- PostgreSQL 16 — `litellm` database (already running)
- FastAPI + httpx — portal-backend spend aggregation service
- Kong 3.6 — new admin routes for key management and spend report

**Storage**: PostgreSQL `litellm` database — `LiteLLM_SpendLogs`, `LiteLLM_VerificationToken` (LiteLLM auto-creates on first start with DATABASE_URL)

**Testing**: `pytest` + `httpx` for portal-backend unit tests; curl-based smoke tests for end-to-end

**Target Platform**: Linux container (OrbStack + Docker Compose, `core` profile)

**Performance Goals**: `GET /v1/spend` ≤ 500 ms p95 (SC-003)

**Constraints**:
- LiteLLM port 4000 has no host binding — all admin API access via Kong
- No prompt content in any log, trace, or database column
- `LITELLM_SALT_KEY` must be set before any virtual key is created
- portal-backend must be multi-platform (`linux/amd64` + `linux/arm64`)

**Scale/Scope**: Up to 10,000 active virtual keys; current-month spend aggregation only

## Implementation Tasks

### T-01: Wire LITELLM_SALT_KEY into docker-compose

**File**: `docker-compose.yml`
**Change**: Add `LITELLM_SALT_KEY: ${LITELLM_SALT_KEY}` to the `litellm` service `environment` block.
**Why**: LiteLLM requires this variable to hash and verify virtual key tokens. Without it, `POST /key/generate` either errors or uses an insecure default.
**Acceptance**:
```bash
make restart svc=litellm && \
  curl -s -X POST http://localhost:8080/v1/key/generate \
    -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" \
    -H "Content-Type: application/json" \
    -d '{"key_alias":"test-salt","max_budget":1.0,"budget_duration":"monthly"}' \
  | python3 -c "import sys,json; d=json.load(sys.stdin); assert 'key' in d, d; print('PASS')"
```

### T-03: Build portal-backend FastAPI service

**File**: `services/portal-backend/main.py`

FastAPI application exposing `GET /v1/spend`. Behaviour:
1. Read `Authorization` header from incoming request.
2. Call `GET http://litellm:4000/global/spend/keys` with the same header.
3. Call `GET http://litellm:4000/global/spend/models` with the same header.
4. If either call returns non-2xx: raise HTTP 503 `spend_store_unavailable`.
5. If LiteLLM returns 401 on either call: raise HTTP 401 `unauthorized`.
6. Aggregate into the response shape defined in `contracts/get-v1-spend.md`.
7. Apply `key_id` filter if query parameter is present.

**Code standards**: Type annotations on all signatures, `ruff` zero warnings, `mypy` strict, `httpx.AsyncClient`.

**File**: `services/portal-backend/requirements.txt`
```
fastapi==0.115.0
uvicorn[standard]==0.32.0
httpx==0.27.2
```

**File**: `services/portal-backend/Dockerfile`
- Multi-stage build (builder + runtime)
- Non-root user
- `linux/amd64` + `linux/arm64` targets
- Pinned base image tag

**Acceptance**:
```bash
# Unit test: mock LiteLLM responses, assert aggregation
cd services/portal-backend && python -m pytest tests/ -v

# Integration: portal-backend container running, LiteLLM running
curl -s http://localhost:8092/v1/spend \
  -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" | python3 -c "
import sys, json
d = json.load(sys.stdin)
assert 'total_spend_usd' in d, d
assert 'by_model' in d, d
assert 'by_key' in d, d
print('PASS')
"
```

### T-05: Add Kong route for GET /v1/spend

**File**: `services/kong/kong.yml`
**Change**: Add a `portal-backend` service pointing to `http://portal-backend:8092`, with route:
- `GET /v1/spend` → `/v1/spend` (strip_path: false)
- `key-auth` plugin: accepts consumer key (so Kong validates auth header is a known credential) OR pass-through to portal-backend's own master-key validation.

**Design note**: Portal-backend validates the Authorization header against LiteLLM — if LiteLLM rejects it, portal-backend returns 401. Kong key-auth here enforces that only registered consumers can call the endpoint at the Kong layer (defence in depth). Since the master key is not a Kong consumer credential, this route uses a **request-termination** plugin check or forwards without Kong key-auth (portal-backend handles auth). Use the simpler approach: no Kong key-auth on this route; portal-backend's proxied auth is sufficient.

**Acceptance**:
```bash
# Unauthenticated → 401
curl -s -o /dev/null -w '%{http_code}' http://localhost:8080/v1/spend
# Expected: 401

# Master key → 200
curl -s -o /dev/null -w '%{http_code}' http://localhost:8080/v1/spend \
  -H "Authorization: Bearer ${LITELLM_MASTER_KEY}"
# Expected: 200
```

### T-07: Extend smoke-test.sh with spend probes

**File**: `scripts/smoke-test.sh`
**Add these probes** after the existing fallback routing probes:

```bash
# Probe: GET /v1/spend unauthenticated → 401
probe "GET /v1/spend — unauthenticated" "${KONG}/v1/spend" 401

# Probe: GET /v1/spend with master key → 200 + required fields
if [[ -n "${LITELLM_MASTER_KEY:-}" ]]; then
  spend_resp=$(curl -s --max-time "$TIMEOUT" \
    -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" \
    "${KONG}/v1/spend")
  if echo "$spend_resp" | python3 -c "
import sys,json
d=json.load(sys.stdin)
assert 'total_spend_usd' in d
assert 'by_model' in d
assert 'by_key' in d
" 2>/dev/null; then
    ok "GET /v1/spend — master key returns required fields"
  else
    fail "GET /v1/spend — response missing required fields: ${spend_resp}"
  fi
else
  printf '[SKIP]    GET /v1/spend — master key probe (LITELLM_MASTER_KEY not set)\n'
fi

# Probe: POST /v1/key/generate with master key → 200
if [[ -n "${LITELLM_MASTER_KEY:-}" ]]; then
  key_resp=$(curl -s -o /dev/null -w '%{http_code}' \
    --max-time "$TIMEOUT" \
    -X POST \
    -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" \
    -H "Content-Type: application/json" \
    -d '{"key_alias":"smoke-key","max_budget":0.001,"budget_duration":"monthly"}' \
    "${KONG}/v1/key/generate")
  if [[ "$key_resp" == "200" ]]; then
    ok "POST /v1/key/generate — master key creates key (HTTP 200)"
  else
    fail "POST /v1/key/generate — expected 200, got ${key_resp}"
  fi
else
  printf '[SKIP]    POST /v1/key/generate probe (LITELLM_MASTER_KEY not set)\n'
fi
```

**Acceptance**: `make smoke` passes all probes including new spend probes.
