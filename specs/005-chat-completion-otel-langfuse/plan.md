# Implementation Plan: Chat Completion with Observability

**Branch**: `005-chat-completion-otel-langfuse` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-chat-completion-otel-langfuse/spec.md`

## Summary

Activate and validate the `POST /v1/chat/completions` inference path through the existing Kong → LiteLLM chain. LiteLLM's already-configured `arize_phoenix` and `langfuse` callbacks handle all observability automatically. New work: add Phoenix Arize and Langfuse to `docker-compose.yml` (obs profile) so callbacks have a live destination, extend LiteLLM config to forward request metadata (`team`, `request_id`, `prompt_name`) to both sinks, confirm LiteLLM's built-in 400 validation for unknown models satisfies SC-004, add a chat-completion contract test, and update the smoke test with a POST probe.

## Technical Context

**Language/Version**: Python 3.11 (contract test only); YAML config updates

**Primary Dependencies**: LiteLLM main-v1.52.0 (chat proxy + callbacks), Kong 3.6 (gateway), Phoenix Arize latest (LLM traces), Langfuse 3 (prompt-linked traces)

**Storage**: PostgreSQL `phoenix` and `langfuse` databases (created by existing `scripts/init-db.sql`)

**Testing**: pytest 7.x (contract tests), bash (smoke-test.sh)

**Target Platform**: Linux server / Docker Compose; `core` profile for basic smoke; `obs` profile for full SC-002/SC-003 verification

**Performance Goals**: p95 < 5 s end-to-end at Kong gateway; p99 < 50 ms for model-validation 400 errors (no upstream call)

**Constraints**: Phoenix image pinned to `:latest` (constitution §3 exception); Langfuse image `langfuse/langfuse:3`; no custom OTel instrumentation code — callbacks handle everything

**Scale/Scope**: 11 models, non-streaming only, single-turn and multi-turn conversations

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|---|---|---|
| I. Request Flow Integrity | **DEFERRED — Guardrails not yet implemented** | Currently: Kong :8080 → LiteLLM :4000 direct. Constitution requires Guardrails :8088 as intermediary for inference. Guardrails is a future feature (safety profile). The Kong `/v1` route will be updated to point to Guardrails when that service is introduced. Exception documented in Complexity Tracking. |
| II. Prompt Content Ephemeral | **PASS** | LiteLLM callbacks log only metadata (token counts, model name, latency). Prompt/response text never written to Phoenix, Langfuse, Loki, or PostgreSQL. Only `llm.token_count.*` and `llm.model_name` OpenInference attributes are stored. |
| III. OpenAI API Compatibility | **PASS** | `POST /v1/chat/completions` is the exact OpenAI chat endpoint path and schema. Response fields map to OpenAI's `choices[0].message.content`, `usage.*_tokens`, `choices[0].finish_reason`. |
| IV. Defence in Depth | **PARTIAL** | Kong key-auth (layer 1) ✅. OPA and Guardrails are future features — their absence is noted; they will be added without breaking this feature's contract. |
| V. Falsifiable Acceptance Criteria | **PASS** | SC-001 through SC-006 are curl/query-verifiable. SC-002/SC-003 require obs profile (Phoenix + Langfuse). |
