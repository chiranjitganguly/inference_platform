# Implementation Plan: Streaming Chat Completions

**Branch**: `006-streaming-sse` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/006-streaming-sse/spec.md`

## Summary

Activate `stream: true` on the existing `POST /v1/chat/completions` endpoint via LiteLLM's native SSE pass-through. New work: explicitly configure the LiteLLM response cache to skip streaming requests, add a streaming contract test file, extend the smoke test with a TTFT probe (`curl -w '%{time_starttransfer}'`), and update the OpenAPI contract to cover the `text/event-stream` response path. No new services, no custom middleware — LiteLLM v1.52 streams the upstream SSE response directly; Kong 3.6 forwards chunked responses without buffering; the `arize_phoenix` callback already closes the Phoenix span at stream completion.

## Technical Context

**Language/Version**: Python 3.11 (contract tests only); YAML config updates

**Primary Dependencies**: LiteLLM main-v1.52.0 (native SSE pass-through), Kong 3.6 (transparent chunked proxy), Prometheus (TTFT via `litellm_request_latency_seconds` histogram), Phoenix Arize (single span per stream), Langfuse (trace created at stream end)

**Storage**: No new storage. LiteLLM Redis cache bypassed for streaming requests via `cache_params.supported_call_types`.

**Testing**: pytest 7.x (contract tests), bash (smoke-test.sh TTFT probe)

**Target Platform**: Linux server / Docker Compose; `core` profile for basic streaming smoke; `obs` profile for span/trace verification

**Performance Goals**: TTFT p95 < 2 s measured at Kong :8080 gateway; model-validation 400 errors p99 < 50 ms

**Constraints**:
- No custom streaming middleware — LiteLLM's built-in SSE pass-through only
- Cache bypass must be explicit in `config.yaml` (FR-013) — cannot rely on LiteLLM default behaviour
- Kong must not buffer SSE — Kong 3.6 proxy mode passes chunked transfer encoding through without modification by default; no config change required
- Each chunk must be independently parseable JSON (FR-003) — this is LiteLLM's native format; verified by contract tests

**Scale/Scope**: 9 chat models (embedding models do not support streaming); non-streaming path must remain unchanged (zero regressions)

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|---|---|---|
| I. Request Flow Integrity | **DEFERRED — same ordered exception as feature 005** | Kong :8080 → LiteLLM :4000 direct. Guardrails does not yet exist. Streaming adds no new bypass beyond what was already documented in 005. |
| II. Prompt Content Ephemeral | **PASS** | Streaming chunks are forwarded to the caller and never persisted. The Phoenix span records only token counts and latency, not chunk content. Langfuse trace records only metadata. |
| III. OpenAI API Compatibility | **PASS** | `choices[0].delta.content` per chunk and `data: [DONE]` sentinel match the OpenAI streaming format exactly. |
| IV. Defence in Depth | **PARTIAL** | Kong key-auth (layer 1) ✅. OPA and Guardrails are future features — their absence is inherited from 005 and documented. |
| V. Falsifiable Acceptance Criteria | **PASS** | SC-001 (TTFT < 2 s) is measured by `curl -w '%{time_starttransfer}'`. SC-002–SC-005 are verifiable via Phoenix/Langfuse query and `make smoke`. |

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| Kong → LiteLLM direct for streaming | Guardrails does not yet exist | Inherited from feature 005. A streaming stub Guardrails would require custom SSE proxying — a distinct and future feature. |
| Explicit cache bypass config in litellm config.yaml | FR-013 is non-negotiable; default LiteLLM behaviour not guaranteed across patch versions | Relying on implicit default (LiteLLM may cache stream responses in some edge cases) risks cache pollution. |
