Phase 00 — Repository & tooling

Week 1No services

Establish the project repository, environment conventions, and the Makefile command interface all phases depend on. These specs define the structural contract every developer on the team must be able to rely on.

FEAT-00-01 · Project initialisation

P00
▾
As a developer joining the project, I want a single setup command that prepares my environment, so that I can start developing within 5 minutes of cloning the repository.
happy pathFirst-time setup on a clean Mac
GivenOrbStack is installed and running on the developer's Mac
Andthe repository has been cloned
Andno .env file exists
Whenthe developer runs bash scripts/setup-mac.sh
Thenthe script exits with code 0
Anda .env file is created from .env.example
Andthe script outputs the message "OK — edit .env then run: make up-core"
failure pathOrbStack not installed
GivenOrbStack is not installed on the Mac
Whenthe developer runs bash scripts/setup-mac.sh
Thenthe script exits with code 1
Andthe script outputs the message "Install OrbStack: brew install orbstack"
edge caseLow available memory warning
GivenOrbStack is installed and running
Andfree system memory is below 3000 MB
Whenthe developer runs bash scripts/setup-mac.sh
Thenthe script outputs a WARNING message about insufficient memory
Butthe script still exits with code 0 and creates .env
Business rules
setup-mac.sh must never overwrite an existing .env file
.env must be present in .gitignore before any commit is made
All variable names in .env.example must have empty values — no real secrets committed

Phase 01 — Core inference stack

Weeks 2–3profile: core~620 MB

The core inference proxy serving all proprietary LLMs via BYOK. Defines the canonical API surface, caching behaviour, model routing, fallback chains, and cost tracking that all subsequent phases build on.

FEAT-01-01 · Model catalogue

P01
▾
As a developer, I want to query all available models and their capabilities, so that I can select the right model for my use case programmatically.
happy pathList all models
Giventhe platform is running
Anda valid API key is provided in the Authorization header
WhenGET /v1/models is called
Thenthe response status is 200 OK
Andthe response body contains an object with "object": "list"
Andthe data array contains exactly 11 entries (9 LLMs + 2 embedding models)
Andeach entry contains fields: id, owned_by, metadata
Andeach entry's metadata contains: tier, context_window, provider
failure pathMissing API key
Giventhe platform is running
WhenGET /v1/models is called without an Authorization header
Thenthe response status is 401 Unauthorized
Business rules
The catalogue must include models from at least 4 distinct providers: openai, anthropic, google, cohere
Every model must have a tier value of either "standard" or "premium"
Embedding models must have type: embedding and a dimensions value in metadata

FEAT-01-04 · Prompt and response caching

P01
▾
As a platform operator, I want identical prompts to return cached responses, so that repeat queries incur zero LLM cost and sub-10ms latency.
happy pathCache hit on identical request
Givena chat completion request has been made with model gpt-4o-mini and a specific messages array
Whenthe identical request (same model + messages + temperature) is made a second time
Thenthe response is returned in under 10 ms
Andthe Prometheus metric litellm_cache_hit_count increments by 1
happy pathCache miss on different request
Givena chat completion request has been made
Whena request is made with the same model but a different messages array
Thenthe request is forwarded to the cloud LLM provider
Andthe litellm_cache_hit_count metric does NOT increment
edge caseCache TTL expiry
Givena response has been cached
Andthe cache TTL of 3600 seconds has elapsed
Whenthe identical request is made again
Thenthe request is forwarded to the cloud LLM provider (cache miss)
Business rules
Cache key is composed of: model + messages array + temperature value
Cache TTL is 3600 seconds (configurable via REDIS_TTL env var)
Streaming requests (stream: true) are NOT cached

FEAT-01-05 · Model fallback routing

P01
▾
As a platform operator, I want failed requests to automatically retry on a fallback model, so that provider outages are invisible to callers.
happy pathPrimary model failure triggers fallback
Giventhe fallback chain for gpt-4o is configured as [claude-sonnet, gemini-pro]
Andthe OpenAI API returns a 503 error
Whena chat completion request is made with model: gpt-4o
ThenLiteLLM retries the request via claude-sonnet
Andthe response is returned successfully
Andthe response body contains the actual model used
edge caseAll fallbacks exhausted
Givenall models in the fallback chain are unavailable
Whena chat completion request is made
Thenthe response status is 503 Service Unavailable
Andthe error body indicates all providers are unreachable
Business rules
A model is marked as in cooldown after 3 consecutive failures (allowed_fails: 3)
Cooldown period is 60 seconds — model is excluded from routing during this window
A maximum of 2 retries are attempted before moving to next fallback model

Phase 02 — Kong API gateway

Weeks 4–5profile: core

Kong becomes the sole entry point for all client traffic. LiteLLM is network-isolated. These specs define the edge enforcement contract — every capability listed here is enforced before any request reaches the model proxy.

FEAT-02-01 · API key authentication at edge

P02
▾
As a platform operator, I want every request rejected at the gateway unless it carries a valid API key, so that LiteLLM is never exposed to unauthenticated traffic.
happy pathValid key passes gateway
GivenKong is running and a consumer team-ml has key sk-team-ml-abc registered
Whena request is sent to POST http://localhost:8080/v1/chat/completions with Authorization: Bearer sk-team-ml-abc
ThenKong forwards the request to LiteLLM
Andthe response status is 200 OK
failure pathMissing Authorization header
GivenKong is running
Whena request is sent to any /v1/* route without an Authorization header
ThenKong returns 401 Unauthorized
Andthe request is NOT forwarded to LiteLLM
failure pathInvalid API key
GivenKong is running
Whena request is sent with Authorization: Bearer sk-invalid-key-xyz
ThenKong returns 401 Unauthorized
happy pathHealth endpoint bypasses auth
WhenGET http://localhost:8080/health is called without any Authorization header
Thenthe response status is 200 OK
Business rules
LiteLLM port 4000 must NOT be accessible from the Docker host after Phase 02 (remove port mapping)
Kong must be the ONLY entry point — no other service exposes ports to the host except monitoring UIs

FEAT-02-02 · Rate limiting per consumer

P02
▾
As a platform operator, I want each API key limited to a maximum request rate, so that no single consumer can saturate the platform and degrade service for others.
happy pathRequests within rate limit succeed
Giventhe rate limit for consumer team-ml is 10 requests per second
Whenexactly 10 requests are sent within one second using key sk-team-ml-abc
Thenall 10 requests are forwarded and return 200 OK
failure path11th request in same second is throttled
Given10 requests have been sent within the current second
Whenan 11th request is sent within the same second
ThenKong returns 429 Too Many Requests
Andthe response includes a Retry-After header
edge caseRate limit is per consumer, not global
Giventeam-ml has reached its rate limit
Whenteam-data sends a request using its own key in the same second
Thenthe team-data request succeeds with 200 OK
Business rules
Rate limit thresholds: second=10, minute=300, hour=10000 per consumer key
Rate limit counters are stored in Redis — shared across all Kong instances

FEAT-02-03 · Request correlation

P02
▾
As a platform operator, I want every request assigned a unique ID at the gateway, so that I can trace a request through all services using a single ID.
happy pathX-Request-ID generated and returned
Whenany authenticated request passes through Kong
Thenthe response includes an X-Request-ID header containing a UUID
Andthe same UUID is forwarded to LiteLLM in the request headers
edge caseClient-provided X-Request-ID is overridden
Givena client sends a request with X-Request-ID: client-provided-id
WhenKong processes the request
ThenKong replaces it with a Kong-generated UUID

FEAT-02-04 · Request size enforcement

P02
▾
As a platform operator, I want oversized requests rejected at the gateway, so that context-stuffing attacks cannot overload the model proxy.
failure pathRequest body exceeds 10 MB

Phase 03 — Full API surface

Weeks 6–8profile: core

Expand the API beyond basic chat completion to cover all interaction modalities the platform must support. These specs define the complete API contract for batch, multimodal, function calling, structured output, and WebSocket streaming.

FEAT-03-01 · Asynchronous batch inference

P03
▾
As a data scientist, I want to submit thousands of inference requests as a single async job and retrieve results when complete, so that I can run large-scale evaluation and annotation pipelines without managing concurrency.
happy pathJob submission and status polling
Givena valid API key is provided
WhenPOST /v1/batch/jobs is called with a requests array of 100 items
Thenthe response status is 202 Accepted
Andthe response body contains a unique job_id
Andthe response body contains status: "queued"
AndGET /v1/batch/jobs/{job_id} returns the job status without blocking
happy pathJob completion and result download
Givena batch job with 100 items has been submitted
Andsufficient time has elapsed for all items to be processed
WhenGET /v1/batch/jobs/{job_id}/results is called
Thenthe response is a JSONL stream with exactly 100 lines
Andeach line contains custom_id, status, and response fields
Andevery custom_id corresponds to an item in the original submission
edge caseConcurrency limit enforced
GivenMAX_CONCURRENT is set to 4
Whena batch job with 100 items is processing

edge caseResults for unknown job_id
WhenGET /v1/batch/jobs/nonexistent-id/results is called

Batch job results are retained for 24 hours after completion, then automatically deleted

Phase 05 — Guardrails & privacy

Weeks 12–14profile: core + safety

Every inference request passes through a pre and post pipeline. These specs define what must be detected, what must be blocked, what must be redacted, and how the audit trail must behave. Privacy compliance is a hard constraint — prompt content is never persisted.

FEAT-05-01 · PII detection and redaction

P05
▾
As a data protection officer, I want personally identifiable information automatically redacted from prompts before they reach any LLM provider, so that user data is protected regardless of which application sends it.
happy pathEmail address redacted before forwarding
Giventhe safety profile is active and Presidio is running
Whena request is made with a message containing an email address (e.g. john@example.com)
Thenthe email address is replaced with <REDACTED> before the request reaches LiteLLM
Andthe audit log entry records pii_redacted: 1
Andthe audit log entry does NOT contain the original email address
happy pathMultiple PII types redacted in one request
Givena message contains an email, a phone number, and a credit card number
Whenthe request is processed by the guardrails service
Thenall three entities are redacted
Andthe audit log records pii_redacted: 3
happy pathNo PII — request passes through unchanged
Givena message contains no detectable PII
Whenthe request is processed
Thenthe message content is forwarded to LiteLLM unchanged
Andthe audit log records pii_redacted: 0
Business rules
PII detection runs on ALL message content fields before any model call — no exceptions
Detection confidence threshold for redaction is 0.80 — entities below this score are not redacted
Redaction strategy is replace with literal string <REDACTED> — not masking or hashing
PII detection latency must not add more than 30ms to the total request latency (measured on local Mac)

FEAT-05-02 · Prompt injection blocking

P05
▾
As a platform security officer, I want adversarial prompts designed to hijack model behaviour automatically blocked, so that the platform cannot be used to bypass application-level guardrails.
failure pathPrompt injection attempt blocked
Giventhe LLM Guard PromptInjection scanner is active with threshold 0.92
Whena request is made with a message designed to override the system prompt or extract instructions
Thenthe guardrails service returns 400 Bad Request
Andthe response body contains error: "request_blocked" and a detail field identifying the PromptInjection scanner
Andthe request is NOT forwarded to LiteLLM
happy pathLegitimate requests not blocked
Givena normal, non-adversarial user message
Whenthe request is processed by the PromptInjection scanner
Thenthe scanner returns is_valid: true
Andthe request proceeds to LiteLLM

FEAT-05-03 · Toxicity and content filtering

P05
▾
As a compliance officer, I want both harmful input prompts and harmful model outputs automatically blocked, so that the platform cannot be used to generate or process content that violates company policy.
