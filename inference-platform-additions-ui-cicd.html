Phase 09 — Unified platform UI

Weeks 23–26Next.js 15 · React 19 · Tailwind v4~120 MB additional

Key screen specifications

▾
Dashboard — /dashboard
Layout: 4-column KPI strip + 2-column charts + model health grid

KPI strip (auto-refresh every 15s via TanStack Query):
  · Platform status: HEALTHY / DEGRADED / DOWN  (derived from /health)
  · TTFT p95:  live value from Prometheus query
  · Error rate %: rate(errors[5m])/rate(total[5m])
  · Active requests: sum(vllm:num_requests_running) or litellm in_flight

Charts row:
  · Left (60%): RPSTimeSeries — stacked by model, last 1h, 6h, 24h toggle
  · Right (40%): SpendBreakdown — pie by model + stat "Spend today $X.XX"

Model health grid:
  · One card per model — status dot (green/amber/red), provider badge,
    avg latency, fallback chain indicator, last-used timestamp
  · Click → /models/{id}  for full detail
Metrics — /metrics
Time range picker: 1h | 6h | 24h | 7d (default 6h)

Row 1 — Latency:
  · TTFTHistogram: heatmap by model, p50/p95/p99 overlaid lines
  · Provider latency box plot: OpenAI vs Anthropic vs Google vs Cohere

Row 2 — Traffic:
  · RPS time series (all models stacked)
  · Cache hit rate time series (target line at 30%)

Row 3 — Errors + Guardrails:
  · Error rate by type (4xx, 5xx, guardrail blocks)
  · PII entities redacted per hour (bar chart)

Row 4 — Cost:
  · Spend per model per day (grouped bar, last 7 days)
  · Budget utilisation table: key alias | spend | budget | % | status badge
Endpoints — /endpoints
Embedded Swagger UI iframe pointed at http://localhost:8090 (portal-backend OpenAPI)
PLUS platform-native additions:
  · Live endpoint health table: path | method | avg latency | error rate | requests today
  · Each row expandable → last 5 requests with status codes
  · WebSocket status indicator (ws://localhost:8080/ws/... connection test)
Keys — /keys
Table: key alias | models allowed | budget | spend | % used | expires | actions
Actions: copy key | edit budget | revoke
Budget bar: green <50% | amber <80% | red >80%
"Create new key" button → /keys/new form with Zod-validated fields:
  · models (multiselect from catalogue)
  · max_budget (number, $)
  · tpm_limit, rpm_limit
  · expiry (date picker)
  · key alias (slug validation)
Server Action: POST to LiteLLM /key/generate → toast on success
Audit log — /audit
Filters: date range | event_type (REQUEST|RESPONSE|BLOCKED_PRE|BLOCKED_POST) | model | key hash
Table (virtualised scroll): timestamp | event_type | request_id | model | pii_redacted | scanner_blocked
Row click → drawer with full audit entry JSON
Export button: download filtered log as JSONL
RULE: Raw prompt text never shown — this view is metadata only

FEAT-09-01 · Platform status dashboard

P09
▾
As a platform operator, I want a single screen showing the real-time health of every service and model, so that I can identify degradation in under 5 seconds without opening Grafana or running curl commands.
happy pathAll services healthy — dashboard shows green
Givena user with any authenticated role opens the platform UI
Whenthe /dashboard page loads
Thena "Platform status: HEALTHY" indicator is visible in the header
Andall 9 model health cards show a green status indicator
AndTTFT p95 and error rate KPI values are visible and non-zero after at least one request has been made
failure pathDegraded provider reflected in UI
Givenone LLM provider's API is returning errors and LiteLLM has placed it in cooldown
Whenthe dashboard auto-refreshes (within 15 seconds)
Thenthe affected model card shows an amber or red status indicator
Andthe platform status header shows "DEGRADED"
Andactive fallback chains are listed on the affected model card
edge caseDashboard auto-refreshes without user interaction
Giventhe user has had the dashboard open for more than 15 seconds
Whenno user interaction occurs
Thenall KPI values and status indicators are silently updated with fresh data
Andno full page reload occurs — only the data changes
Business rules
Dashboard data must never be more than 30 seconds stale when the tab is in focus
Platform status is HEALTHY only if all configured services are reachable and all models have at least one healthy provider
A user can see the dashboard regardless of their role — visibility is not role-restricted

Phase 10 — CI/CD pipelines

Weeks 27–30GitHub Actions · Cloud Build · ArgoCD · HelmLocal K3s → GCP GKE

Automated build, test, security scan, and deployment pipelines for every service in the platform. Local developer loop via Skaffold targeting K3s. Production deployments via GitOps on GKE. Container images signed with Cosign. No manual deployments after this phase.

.github/workflows/ci.yml — pull request pipeline

▾
yaml
name: CI — build, test, scan

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

permissions:
  contents: read
  security-events: write        # For SARIF upload
  id-token: write               # For OIDC to GCP

env:
  REGISTRY: us-central1-docker.pkg.dev
  PROJECT:  ${{ secrets.GCP_PROJECT_ID }}
  REPO:     inference-platform

jobs:

  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Lint Python services
        run: |
          pip install ruff mypy
          ruff check services/guardrails/ services/batch-worker/ services/portal-backend/
          mypy services/guardrails/ --ignore-missing-imports

      - name: Lint TypeScript (platform-ui)
        working-directory: services/platform-ui
        run: |
          pnpm install --frozen-lockfile
          pnpm tsc --noEmit
          pnpm lint

      - name: Run smoke tests (mock mode)
        run: bash scripts/smoke-test.sh http://localhost:4000 --mock

  build-and-push:
    needs: lint-and-test
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [litellm, kong, guardrails, batch-worker, portal-backend, platform-ui]
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to GCP
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.GCP_WI_PROVIDER }}
          service_account:            ${{ secrets.GCP_SA_EMAIL }}

      - name: Configure Docker for Artifact Registry
        run: gcloud auth configure-docker ${{ env.REGISTRY }} --quiet

      - name: Build and push with Buildx
        uses: docker/build-push-action@v6
        with:
          context:   ./services/${{ matrix.service }}
          platforms: linux/amd64,linux/arm64
          push:      ${{ github.event_name == 'push' }}
          tags:      ${{ env.REGISTRY }}/${{ env.PROJECT }}/${{ env.REPO }}/${{ matrix.service }}:${{ github.sha }}
          cache-from: type=gha
          cache-to:   type=gha,mode=max
          provenance: true
          sbom:       true

  scan:
    needs: build-and-push
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [litellm, kong, guardrails, batch-worker, portal-backend, platform-ui]
    steps:
      - name: Trivy vulnerability scan
        uses: aquasecurity/trivy-action@v0.28.0
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.PROJECT }}/${{ env.REPO }}/${{ matrix.service }}:${{ github.sha }}
          format:    sarif
          output:    trivy-${{ matrix.service }}.sarif
          severity:  CRITICAL,HIGH
          exit-code: "1"            # Fail pipeline on CRITICAL CVEs

      - name: Upload SARIF to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-${{ matrix.service }}.sarif

  sign:
    needs: scan
    if: github.event_name == 'push'
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [litellm, kong, guardrails, batch-worker, portal-backend, platform-ui]
    steps:
      - uses: sigstore/cosign-installer@v3

      - name: Sign container image (keyless OIDC)
        run: |
          cosign sign --yes \
            ${{ env.REGISTRY }}/${{ env.PROJECT }}/${{ env.REPO }}/${{ matrix.service }}:${{ github.sha }}

Acceptance criteria — tech

▾
  • Opening a PR triggers ci.yml — lint, build, Trivy scan, and Cosign sign run automatically
  • Trivy scan: CRITICAL CVE in any image fails the pipeline — PR cannot be merged
  • Push to main triggers cd-gcp-dev.yml — images pushed to Artifact Registry and Cloud Deploy release created
  • Cosign signature verified before any image is deployed to GKE — unsigned images rejected
  • skaffold dev on Mac: file change in guardrails/main.py triggers hot-sync to K3s within 5 seconds
  • ArgoCD UI shows all Applications in Synced state after a clean deploy
  • ArgoCD self-heals: manual kubectl patch to litellm deployment is reverted within 3 minutes
  • Cloud Deploy canary: 10% traffic to new version, metrics checked, auto-promote or rollback
  • gcloud deploy rollback one command returns prod to previous release in under 5 minutes
  • GitHub Actions uses OIDC to GCP — zero long-lived service account keys in repository secrets
  • All images are multi-platform (linux/amd64 + linux/arm64) — verified with docker buildx imagetools inspect

As a developer, I want every pull request automatically built, tested, and scanned for vulnerabilities, so that I receive feedback within 10 minutes of opening a PR without waiting for a manual review.

FEAT-10-06 · Environment promotion pipeline

P10
▾

Whenthe CI pipeline completes successfully
